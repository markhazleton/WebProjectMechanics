## Overview

This command reviews the **full commit history** of the repository and produces a compelling, evidence-based narrative — a "repo story" — suitable for both **business stakeholders** and **technical audiences**. It draws on commit metadata, contributor patterns, velocity metrics, governance signals, and milestone history generated by the context scripts.

**IMPORTANT**: This command **only produces a narrative document** — it does not make any code changes.

## Prerequisites

- Git repository with commit history (REQUIRED)
- PowerShell 7+ or Bash (for script execution)
- Python 3 (used by the bash script for JSON assembly)
- Constitution at `/.documentation/memory/constitution.md` (optional — used for governance alignment section)

## Scope Options

Parse `$ARGUMENTS` for scope flags:

| Flag | Description |
|------|-------------|
| `--scope=full` | Complete story (default) — all sections |
| `--scope=velocity` | Development pace and throughput focus |
| `--scope=quality` | Quality signals, test metrics, governance |
| `--scope=business` | Business-audience executive summary only |
| `--scope=team` | Contributor patterns and team evolution |
| `--months <n>` | History window in months (default: 12) |
| `--compare-baseline YYYY-MM` | Compare current period against a baseline month |

If no scope specified, default to `--scope=full`.

## Outline

### 1. Generate History Context

Run `{SCRIPT}` to produce `history.json` and parse the JSON output for these top-level sections:

- `audit_parameters` — time window, scope, anonymization settings
- `repo` — repository name, remote URL, default branch
- `summary` — total commits, contributors, first/latest commit dates
- `timeline` — commits grouped by month
- `contributors` — per-contributor commit counts and roles
- `contributor_census` — anonymized role distribution (Lead Architect, Developer A/B/C, etc.)
- `volume_metrics` — lines added/removed, files changed, churn ratio
- `commit_message_metrics` — conventional commit adoption, avg length, prefix distribution
- `test_metrics` — test file counts, test-to-source ratio
- `governance_maturity` — merge commit %, PR-based workflow signals, tag discipline
- `milestones` — tags with dates and messages
- `change_patterns` — most-modified files, hotspot directories
- `technical_signals` — language breakdown, config file presence, CI indicators
- `recent` — last 20 commits for recency context

**Error Handling**:
If the script fails:

- **Git not available**: Inform user that a Git repository is required
- **No commits found**: Report that the repository has no history to analyze
- **Python missing (bash only)**: Suggest running the PowerShell variant or installing Python 3

### 2. Validate Context Quality

Before building the narrative, assess the history data:

| Check | Action if Failed |
|-------|-----------------|
| `summary.total_commits` < 10 | Warn that the story may be thin; proceed with caveats |
| `timeline` has < 2 months | Note limited timeline; focus on depth over trends |
| `contributors` has only 1 entry | Frame as solo-developer journey rather than team narrative |
| `milestones` is empty | Skip milestone arc; note absence of tagged releases |

### 3. Build Narrative Arc

Construct the repo story using the following structure. Every claim **MUST** cite specific numbers from `history.json`.

#### A. Executive Summary (Business Audience)

Write 3–5 paragraphs covering:

- **What this project is** — purpose derived from repo name, README presence, and language mix
- **Scale and maturity** — total commits, contributor count, age of first commit, latest activity
- **Velocity trend** — month-over-month commit patterns from `timeline`, acceleration or deceleration
- **Governance posture** — merge commit %, conventional commit adoption, tag discipline score
- **Key milestones** — tagged releases with dates, framing them as delivery evidence

Use plain language. Avoid jargon. Target a VP/director reading level.

#### B. Technical Deep Dive

For each of the following subsections, provide evidence-backed analysis:

##### Development Velocity

- Commits per month trend (cite `timeline` data)
- Lines added vs removed (cite `volume_metrics`)
- Churn ratio interpretation — high churn = refactoring, low churn = greenfield
- If `--compare-baseline` provided, show delta between baseline and current period

##### Contributor Dynamics

- Role distribution from `contributor_census` (use anonymized roles, not real names)
- Bus factor assessment — what percentage of commits come from top contributor?
- Team growth or contraction over the timeline

##### Quality Signals

- Test-to-source file ratio (cite `test_metrics`)
- Conventional commit adoption rate (cite `commit_message_metrics.conventional_percentage`)
- Commit message quality — average length, prefix diversity

##### Governance & Process Maturity

- Merge commit percentage as PR workflow proxy (cite `governance_maturity`)
- Tag discipline — regular releases vs sporadic
- Branch strategy signals from merge patterns

##### Architecture & Technology

- Language/framework indicators from `technical_signals`
- Configuration maturity — CI, linting, formatting, dependency management
- Hotspot analysis — most-modified files may indicate active development or instability

#### C. Change Pattern Analysis

Using `change_patterns`:

- Identify the top 5 most-modified files and explain what they suggest
- Flag any files with disproportionate churn as potential refactoring candidates or complexity hotspots
- Note directory-level patterns (e.g., "80% of changes in src/ — active development area")

#### D. Milestone Timeline

If `milestones` is non-empty, produce a chronological timeline:

```markdown
| Date | Tag | Description |
|------|-----|-------------|
| ... | ... | ... |
```

Connect milestones to surrounding commit activity — did velocity spike before releases? Drop after?

#### E. Constitution Alignment (Optional)

If `/.documentation/memory/constitution.md` exists:

- Read the constitution principles
- Map governance metrics to relevant principles
- Note areas of strong alignment and gaps
- Frame as "How well does the commit history reflect stated values?"

## Developer FAQ

> Answer these questions from first principles using only facts from `history.json`, `technical_signals`, `change_patterns`, and the current state of the repository. Do not guess. If the data is absent, say so.

Generate a FAQ section targeted at a **developer who is new to this repository**. Each question must be answered with specific, current evidence — not generic advice.

Minimum required questions (add more if the data supports them):

| # | Question |
|---|----------|
| 1 | **What does this project do?** — Summarize purpose from README/repo name/language mix |
| 2 | **What tech stack does it use?** — Languages, frameworks, key dependencies from `technical_signals` |
| 3 | **Where do I start?** — Entry-point files from `change_patterns` most-modified list + README install steps |
| 4 | **How do I run it locally?** — Build/run commands from README, CI config, and config files present |
| 5 | **How do I run the tests?** — Test framework detected from `test_metrics`, test file locations |
| 6 | **What is the branching/PR workflow?** — Derived from `governance_maturity` merge % and branch signals |
| 7 | **Who do I ask when I'm stuck?** — Top contributor role from `contributor_census` (anonymized) |
| 8 | **What areas of the code change most often?** — Top 3 hotspot files/dirs from `change_patterns` |
| 9 | **Are there coding standards I must follow?** — Conventional commit adoption, linting/format config presence |
| 10 | **What version is currently released?** — Latest tag from `milestones`, date, description |

Format each answer as a short paragraph (2–5 sentences) immediately under its heading. Use concrete values from the data (file names, command strings, version tags, percentages).

### 5. Link Story in README

After saving the narrative, update the project's root `README.md` to surface the repo story.

**Locate the README** at the repository root (or the closest parent containing a README).

**Add or update a "Repo Story" entry** in one of the following ways (in order of preference):

1. **If a "Learn More" or similar table exists** — add a row:

   ```markdown
   | Repo story | [repo-story/repo-story-{YYYY-MM-DD}.md](.documentation/repo-story/repo-story-{YYYY-MM-DD}.md) |
   ```

   If a prior repo-story row already exists, replace it with the new dated link.

1. **If no such table exists** — append a brief section before the final License/Contributing section:

   ```markdown
   ## Repo Story

   An evidence-based narrative of this repository's development history, contributor patterns, and architecture:
   [Repo Story ({YYYY-MM-DD})](.documentation/repo-story/repo-story-{YYYY-MM-DD}.md)
   ```

**Rules**:

- Always use the most recent story file (today's date). Remove any older repo-story links in README.
- Use a relative path from the README to the story file.
- Do not create a new heading if a suitable one already exists — add the link there.
